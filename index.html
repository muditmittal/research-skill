How it works

From question to Google Doc
in four phases

Type /research <your question> and the system handles the rest.

01
Dispatch
The question is matched against 20 dispatch patterns. The right subset of agents is selected and spawned in parallel.
INDEX.md
02
Collect
Each agent searches its domain and returns findings in a standardized format — key findings, evidence table, gaps, and a coverage confidence rating.
agents/
03
Synthesize
Coverage confidence check, conflicts resolved via 9-level trust hierarchy, citations renumbered into a unified reference list. Product Design agent synthesizes all findings into a consolidated design implications section.
INDEX.md
04
Publish
Findings-first Google Doc — executive summary, themes, design implications, references, appendix. Follow-ups update the same doc in place; URL never changes.

The agents

Ten lenses on every question

Each agent declares its bias upfront. The synthesis judge knows where to discount and cross-reference.

Official Docs
docs.databricks.com
What Databricks has shipped, documented, and publicly committed to. Covers feature scope, known limitations, API surface, pricing, and cloud differences.
Skews toward shipped & documented — misses unbuilt concepts and the gap between docs and real UX
Community Voice
forums · Reddit · Ideas Portal
Public user sentiment, feature requests, pain points, and praise from community forums, Reddit, and the Ideas Portal.
Skews toward vocal power users — underrepresents silent churners
Internal Intel
Glean · Confluence · Slack
PRDs, UXR archive, PM customer notes, engineering support tickets, CUJ definitions, and prior design work from internal Databricks sources.
Skews toward past internal thinking — may anchor to prior approaches
Competitive
battlecards · CI Confluence
How competitors approach the problem — features, positioning, win/loss patterns, and battlecard intelligence across all product areas.
Skews toward Databricks-favorable framing from battlecards
Quantitative
Logfood · SimpleBricks
Executes actual SQL queries against Logfood — adoption metrics, funnel analysis, CUJ completion scores. Returns real numbers with reproducible queries, not hypotheticals.
Can only measure what's instrumented — zero ≠ no demand
Market Landscape
Gartner · MAD landscape
Broader market context, analyst views (Gartner, Forrester), vendor ecosystem, and the full ~1,200-company MAD landscape catalog.
Analyst views lag 6–18 months — overrepresents funded vendors
Roadmap & Planning
Aha! · Jira · pre-reads
Active and planned work, quarterly commitments, and leadership priorities from pre-reads, Aha!, Jira, and OKR documents.
Skews toward formally planned work — misses grassroots efforts
Foundations
academic · W3C · ISO
Academic grounding, industry standards (W3C, ISO, DMBOK), canonical definitions, and conceptual frameworks from peer-reviewed sources.
Skews theoretical — may over-index ideal-state vs. practical reality
Sales & GTM
BrickBites · SFDC · QBRs
Field signal from AE BrickBites, deal dynamics, win/loss data from SFDC Compete360, and PMM positioning frameworks.
"Sells well" ≠ "works well" — skews toward deal-winning narratives
Product Design
NNGroup · design systems · competitor UX
UX patterns, competitor UI analysis, and design implications. Runs twice — as a researcher in Phase 2, then synthesizes all findings into a consolidated design implications section with an "Explore these ideas" prompt.
May underweight engineering feasibility and business constraints
Why it's different

Built for signal quality,
not just coverage

Most research is a pile of links. This system is designed for synthesis.

📊
Coverage Confidence
Before synthesizing, the system checks how many agents returned substantive findings. Under-researched topics are flagged — not presented as conclusive.
⚖️
9-Level Trust Hierarchy
When agents conflict, a ranked authority chain resolves it mechanically. Logfood data beats battlecards. Customer notes beat analyst reports. Declared upfront, applied consistently.
🔗
Inline Citations
Every factual claim has a [N] reference linking to the source. Citations are renumbered into a unified reference list during synthesis — grouped by internal data, internal docs, and external sources.
🎨
Design Implications
Every output ends with a consolidated design section: user mental model, interaction patterns, IA considerations, UX risks, and an "Explore these ideas" prompt for generating mockups.
🔄
Follow-Up Preservation
Follow-up research surgically updates the existing Google Doc — only changed sections are modified. The URL never changes. No more broken links in Slack or Confluence.
📄
Findings-First Output
Executive summary, themes, conflicts, design implications up front. Research methodology and raw agent outputs in the appendix — because stakeholders care about findings, not process.
🔢
Real Quantitative Data
The Quantitative agent executes actual SQL against Logfood and returns real numbers — not hypothetical queries. Every data point includes the query, time period, and caveats.
🏷️
Shared Tag Taxonomy
All agents tag findings across 5 dimensions: source type, business function, signal quality, topic domain, and access level. Conflict resolution becomes mechanical.
🔧
Any Product Area
Ships with Data Governance defaults. Swap three files for AI/BI, Pipelines, Compute, GenAI, or any other area. The dispatch and synthesis system is topic-agnostic.

Get started

Install in five steps

Requires Claude Code with MCP servers configured. See prerequisites below.

1

Copy the skill to Claude Code

Copy the skill/ folder into Claude Code's docs directory so it can be loaded at research time.

# Run from the repo root
cp -r skill/ ~/.cursor/docs/research/
2

Read INDEX.md before first use

The entry point is INDEX.md. Skim the dispatch table and synthesis rubric so you know what to expect from each research run.

cat ~/.cursor/docs/research/INDEX.md
3

Customize for your product area

The skill ships with Data Governance defaults. If you work on AI/BI, Pipelines, Compute, or another area, follow the 5 steps in CUSTOMIZE.md before running research.

cat ~/.cursor/docs/research/CUSTOMIZE.md
4

Verify your MCP connections

Check that Google Docs, Glean, Databricks (Logfood), Slack, and Jira MCPs are active in Claude Code. If any are missing, the corresponding agent will note a gap — the run won't break, but coverage will be lower. See prerequisites below for access details.

5

Run your first research

/research Should we build an Ontology feature inside Databricks?

Customization

Works for any product area

Ships with Data Governance / Unity Catalog defaults. Three files need replacing for other teams — everything else works as-is.

⚙️ Replace
intel/governance-intel.md
Topic-specific Databricks doc links. Swap with links for your area: AI/BI, Lakeflow, GenAI, Compute, Security, or Apps.
⚙️ Replace
intel/db-data.md
Logfood schemas and workspace URLs. Add your team's tables and fill in Part 2 (workspace access) with your specific environment.
⚙️ Replace
agents/agent-competitive.md
Competitor table. Update the key competitor list for your area (AI/BI, Pipelines, MLOps, GenAI, etc.).
